# 設計判断

後から「なぜこの方式なのか」を判断できるよう、影響の大きい決定を記録します。

## ADR-001: ブラウザ完結のPWAを最初の配信面にする

- 状態: 採用
- 理由: URLだけで試せ、スマートフォンとデスクトップで利用でき、オフライン学習にも向く。
- 代償: OS固有のCodex利用履歴を直接読めず、端末間の進捗同期もない。

## ADR-002: 事実根拠をOpenAI公式Codexドキュメントに限定する

- 状態: 採用
- 理由: Codexは更新頻度が高く、二次資料やClaude固有仕様の転用は誤りを生みやすい。
- 代償: 問題追加時と定期監査に公式資料の再確認が必要になる。

## ADR-003: 難易度と実務価値を別軸で持つ

- 状態: 段階導入中
- 理由: 高度でも頻用する機能と、簡単でも暗記に近い知識を同じ尺度で扱わないため。
- 代償: 旧問題へのメタデータ付与と出題ロジックの調整が必要になる。

## ADR-004: 問題に図解を紐付ける

- 状態: 採用
- 理由: コマンド列、処理順、surface比較、設定階層は文章だけより視覚表現が理解しやすい。
- 代償: 問題変更時に図解との整合性も確認する必要がある。

## ADR-005: 選択肢を表示時にランダム化する

- 状態: 採用
- 理由: 問題データの正解位置を覚えるだけの学習を防ぐ。
- 代償: 表示インデックスと元データの正解を常に対応させ、テストする必要がある。

## ADR-006: 学習データをlocalStorageへ保存する

- 状態: 採用
- 理由: ログインやサーバーなしで、履歴・SRS・途中再開を提供できる。
- 代償: ブラウザデータ削除への備えとしてJSON入出力が必要で、自動同期は提供できない。
